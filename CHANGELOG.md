## v1.0.3 — 2026-07-03

Sprint 9.2 — **Theme Library Foundation.** An architecture sprint, not
a feature sprint: transforms the Theme Engine's hardcoded
`BUILTIN_THEMES` array into a `ThemeRegistry` abstraction layer, so
themes become portable creative assets instead of application data —
groundwork for the future Theme Creator, Theme Export, and Community
Themes, none of which ship in this sprint.

- feat(theme): Sprint 9.2 Theme Library Foundation (build 0051, 2026-07-03)
  - **New `js/themeRegistry.js`.** Single abstraction layer between
    theme storage and `ThemeEngine`, mirroring the `list / find /
    register / validate` shape `storyDestinations.js` already
    established for publishing destinations. Owns the four built-in
    themes (moved here verbatim from `themeEngine.js` — same ids, same
    field values) plus whatever a child imports, merged into one
    catalog behind `get(id)` / `list()` / `getCatalog()`. `ThemeEngine`
    never learns which collection a theme came from.
  - **`.vtheme` package format.** Plain, human-readable JSON (not
    zipped or compressed, so it can grow into a packaged format later
    without an API change): `{manifest: {id, name, version, author,
    description, category, tags, thumbnail, createdDate, updatedDate,
    minStudioVersion}, theme: {...}, assets: {}}`. `theme` is the exact
    flat shape the renderer has always consumed — nothing downstream
    changed.
  - **Validation, never crashes.** `ThemeRegistry.validatePackage`
    checks JSON structure, required manifest fields, id format, and
    `minStudioVersion` compatibility, returning friendly problem
    strings instead of throwing. A corrupt or incompatible `.vtheme`
    shows an `alert()` (same UX as a failed project open) and leaves
    the Theme Library untouched.
  - **Duplicate handling.** Importing a `.vtheme` whose id already
    exists prompts **Replace Existing Theme / Keep Both / Cancel** (new
    `themeImportConflictModal`, same shape as the existing
    restore-session prompt). Keep Both appends `-copy` (bumping to
    `-copy-2` etc. if that's also taken); Replace overwrites in place;
    both persist to `localStorage` immediately.
  - **Theme Library UI.** The existing Theme Picker modal (Sprint
    3.3.1) is restructured into **Official Themes** / **Imported
    Themes** sections + a **+ Import Theme** button, rendered from
    `ThemeRegistry.getCatalog()`. No search, no filters, no delete, no
    rename, no drag-and-drop — intentionally out of scope. Card
    click-to-apply behaviour is unchanged.
  - **Backward compatible by construction.** `ThemeEngine.getTheme /
    getAllThemes / getActiveTheme / applyTheme / registerTheme` all
    keep their exact pre-9.2 signatures; only the storage underneath
    changed. Projects still store only `themeId` (unchanged — never an
    embedded theme object). Verified in headless Chromium: a legacy
    project payload referencing `theme:"fun-comic"` opens with zero
    errors and resolves the exact original frame colour (`#FFD700`);
    importing a theme, applying it, re-importing to trigger the
    conflict prompt, choosing Keep Both, and reloading the page all
    behave correctly with imported themes persisting across reload;
    Theme Designer renders correctly against a single-variant,
    zero-decoration imported theme (no crash).
  - **Architecture ready for what's next without another redesign.**
    Manifest metadata (author, version, category, tags) and
    `source: 'official'|'imported'` tagging already exist per theme, so
    Theme Creator, Theme Editor, Theme Export, Duplicate Theme,
    Community Themes, Official Theme Updates, and eventual Theme
    Marketplace / Versioning / Synchronization are additive — new
    sources feeding the same registry, not a new architecture.

## v1.0.2 — 2026-07-01

Sprint 9.1 — **Trust & Publishing Rework.** A product rework driven by
real user testing after v1.0.1: the editor still felt softer than the
published output on plain DPR=1 displays, Theme changes weren't reaching
every page, and Export terminology was still lingering. Every observable
issue is fixed, and every acceptance criterion is demonstrated by
committed screenshots in `docs/9.1-evidence/`.

- feat(publish, theme, editor): Sprint 9.1 Trust & Publishing Rework (build 0050, 2026-07-01)
  - **WYSIWYE investigation (9.1.1) — real root cause identified.** The Sprint 9.0.2 DPR fix helped retina users but the DPR=1 editor was still visibly softer than the published Book. Empirical measurement in headless Chromium at `deviceScaleFactor=1` isolated the culprit: `#previewCanvas` was CSS-locked to 400–500 CSS px width, forcing the browser to downsample the 1080-wide canvas backing by 2.70×, while the Chrome PDF viewer natively paints the same source at ~720 CSS px (1.50× downsample). See `RENDER-AUDIT-9.1.1.md` for the full findings.
  - **Render parity fix (9.1.2) — fluid preview canvas.** `#previewCanvas` is now CSS-fluid (`max-width:100%`, `max-height:100%`, `aspect-ratio:4/5`). The canvas fills every available CSS pixel above the timeline, so on a 1440-wide viewport the editor renders at 620 CSS px (**1.74× downsample**, down from 2.70×) — inside the same 1.5–1.8× band the PDF viewer uses. Media-query hardcoded canvas widths at 1920 / 1600 / 1440 / 1366 / 1280 px removed; the sidebars still trim per breakpoint so the preview column stays generous. **Pixel parity confirmed:** the editor canvas rendered at dpr:1 and the Story Carousel Portrait PNG are byte-identical for the same slide — both hash to `SHA-256 = 86e6910993d29932` — see `docs/9.1-evidence/editor-canvas.png` + `carousel-portrait-page-2.png`.
  - **Theme Designer global behaviour (9.1.3) — critical propagation fix.** `SlideRenderer.buildPayload` now stamps the RESOLVED theme (`ThemeEngine.resolveTheme()` — with `themeOptions.typography` + `themeOptions.colours` layered on) instead of the base theme. Before this, Typography / Colours changes updated `themeOptions` but never reached the renderer through the payload, so font-family / colour changes silently didn't propagate to thumbnails, the read canvas, or the publish pipeline. Also fixed: `ThemeEngine._invalidateThumbnails` now clears every slide's cached thumbnail — including cover / hook / end pages that render without an uploaded image — so the sidebar reflects a Theme change on every page. New intro note at the top of the Theme Designer tab reinforces the rule: "Theme is the story's global design system. Every change here updates every page instantly. No selection needed." **Verified end-to-end:** a `themeOptions.typography.color` change flips every thumbnail SHA-256 in the sidebar (cover, story, end) without any object or page selection — see `docs/9.1-evidence/theme-before.png` + `theme-after.png` + `theme-thumbnails.png` + `theme-typography.png`.
  - **Publishing Language & Page Management (9.1.4) — Export is gone.** The hidden `#exportBtn` is removed from `index.html`; its dormant handler + null-guard are removed from `js/app.js`. Publish is the editor's only publishing action. The page context menu is rebuilt around the six actions children reach for: `📖 Publish This Page`, `📋 Duplicate Page`, `⬆ Move Page Up`, `⬇ Move Page Down`, `➕ Add Blank Page After`, `🗑 Delete Page`. Rename Page is removed. Move Page Up / Down wire through `PageOps.reorderPage` and honour `PageOps.canMove` so Cover / End pages stay anchored — the same rules the drag-and-drop reorder already respects. `PublishStudio.open({slides:[oneSlide]})` accepts a single-page slice; a new `_slides()` helper resolves the scope so Read → Almost Ready → Destination → Publishing → Celebration all reuse the full pipeline. **Verified:** a Publish This Page → Story Book / Digital PDF flow from a story page emits a PDF with exactly 1 embedded JPEG (~30 KB), not 4 — see `docs/9.1-evidence/page-context-menu.png` + `publish-button.png`.
  - **Validation packet (9.1.5).** `docs/9.1-evidence/` ships 11 screenshots + assets that demonstrate every acceptance criterion, alongside `docs/9.1-evidence/README.md` explaining each file. Cross-surface hashes recorded in the sprint commits (WYSIWYE byte-identity, thumbnail regeneration proof, single-page publish JPEG count).
  - **Architecture preserved.** No project format change, no Theme Engine schema change, no renderer pipeline redesign. Every fix ships through the same `SlideRenderer.buildPayload → SlideRenderer.render` path the editor already uses. Saved projects from v1.0.1 round-trip byte-identically.

## v1.0.1 — 2026-07-01

Sprint 9.0 — **WYSIWYE & Story Publishing Experience.** Two changes ship in this release. First, the editor now renders at the device's full pixel density, so what a child sees in the editor is exactly what they get when they publish. Second, publishing is no longer a technical action that produces a "PDF" or an "export" — children choose a **Story Destination** (Story Book, Story Carousel, Story Reel), software picks the file format, and the child walks away with a real deliverable.

- feat(publish): Sprint 9.0 WYSIWYE & Story Destinations (build 0049, 2026-07-01)
  - **Rendering audit (9.0.1).** Investigated the three rendering pipelines (editor / Publish Studio Read / PDF export) and documented every place they diverge. All three share the same `SlideRenderer.render` code path with identical anti-aliasing / smoothing settings; the only divergence was that the editor + Read canvases were CSS-downscaled from a 1080 × 1350 backing store without devicePixelRatio compensation, while the PDF preserved the full 1080 × 1350 bitmap. See `RENDER-AUDIT-9.0.1.md` for the full findings.
  - **Editor & Publish render parity (9.0.2).** `SlideRenderer.init(canvas, opts)` now takes an optional `{dpr}` in `opts`, defaulting to `window.devicePixelRatio || 1`. Backing store becomes `W*dpr × H*dpr` with `ctx.setTransform(dpr, 0, 0, dpr, 0, 0)` so downstream renderer code keeps drawing in canonical 1080 × 1350 coordinates. Verified in headless Chromium: at DPR = 2 the editor canvas backing jumps to 2160 × 2700 (4× the pixel data of the DPR = 1 baseline) while the CSS display size stays fixed — the editor now ships ~2.8× more PNG bytes of detail. New `SlideRenderer.getCanvasSize()` exposes the canonical logical size so hit-testing (mouse coords) uses logical pixels regardless of DPR. Export paths (`PublishStudio._renderNextPage`, thumbnails) explicitly pass `{dpr: 1}` so their `toDataURL` output stays 1080 × 1350 flat.
  - **Story Destinations — Book + Carousel (9.0.3).** New `js/storyDestinations.js` registry replaces the hardcoded PDF-only publish loop. Destinations expose `id / label / glyph / tagline / formats[] / createCanvas / renderPage / encodePage / finish`; the Publishing stage iterates the same way regardless of destination. **Story Book** ships Digital PDF (1080 × 1350 JPEG in a 540 × 675 pt page) + Print-ready PDF (1620 × 2025 JPEG for higher fidelity). **Story Carousel** ships Instagram Portrait (1080 × 1350 PNG) + Instagram Square (1080 × 1080 centre-cropped PNG). Multi-page carousels bundle into a **ZIP archive** via a new hand-rolled `js/zipWriter.js` (STORED-only PKZip, ~140 LOC, deterministic MS-DOS timestamps so repeated exports of the same content produce byte-identical ZIPs). New DESTINATION stage sits between Almost Ready and Publishing; the Almost Ready primary button now reads **Choose Story Destination** instead of Publish My Book.
  - **Publishing architecture — Reel-ready (9.0.4).** `StoryDestinations.register(destination)` + `validate(destination)` let future plugins push destinations into the registry from outside `storyDestinations.js` — no PublishStudio edits, no shell changes. Validator reports missing fields / hooks individually, rejects duplicate ids, and treats `comingSoon` destinations as inert (they may skip render hooks). Verified end-to-end: a `plaintext` "Story Transcript" plugin registered from a test script appears in the picker and renders through the full Publish loop, producing a 69-byte text/plain blob with page-by-page content. **Story Reel** ships as a Coming Soon card with its formats declared (Instagram Reel 1080 × 1920); selecting Reel disables Continue and surfaces a friendly "Coming Soon" note so no partial file is ever emitted. The interface is ready for the MP4 renderer to drop in during a future sprint.
  - **Celebration & validation (9.0.5).** Destination-aware Celebration copy: Story Book → "Get My Book" / "Your story is now a real book!"; Story Carousel → "Download Images" / "Your story is now a shareable carousel!" (or "Download Image" for single-page). The download filename + button glyph now come from the destination's `finish()` metadata so every future destination gets celebration for free. **Visual parity confirmed at the pixel level:** the editor canvas rendered at dpr:1 and the Carousel Portrait PNG produce byte-identical output (both 61,599 bytes, SHA-256 = `9cfbe2fe08d30291`). WYSIWYE now holds byte-identically across the editor and the shipped Carousel output. The Book PDF differs by design (JPEG's inherent lossy compression = 45 KB vs the PNG's 61 KB) but draws from the same source pixels.
  - **Architecture preserved.** No project format change, no Theme Engine change, no Scene Engine change, no renderer pipeline redesign. Everything rides on the same `SlideRenderer.buildPayload → SlideRenderer.render` path the editor already uses. Story Reel plugs in later without touching PublishStudio.
  - Verified end-to-end in headless Chromium against a 2-page book: Book/Digital → 66 KB PDF; Carousel/Portrait → 106 KB ZIP with two PNG pages; Carousel/Square → 54 KB ZIP; Reel → Continue disabled + Coming Soon message visible. Registry extensibility check: a plaintext plugin registered from outside `storyDestinations.js` renders through the same loop.

- fix(fidelity): Sprint 6.3 Rendering Fidelity Audit & Fix (build 0035, 2026-06-29)
  - **Audit findings.** The candle-sketch test case showed visible degradation. Root causes:
    1. `ProjectManager.imageToDataURL` re-encoded every saved image as **JPEG at 0.92** — measurable artefacts on pencil texture, contrast, and edges, compounding on every Save → Reload.
    2. The main upload pipeline in `app.js` decoded the file directly via a blob URL and **never set `slide._imageDataURL`**, so the lossy fallback was the *only* path the image took before persistence.
    3. Chromium defaults `imageSmoothingQuality` to **`'low'`** — bilinear filtering. The renderer downscales high-resolution scans (e.g. 1500×2000) into the image-holder rect in a single `drawImage` call; with `'low'` quality the fine pencil strokes smudge.
