# CLAUDE.md

Canonical project context for VihuStudio. Read this first before making any change.

## Product Vision

- Preserve what is real.
- Beautify originals rather than replacing them.
- VihuStudio is a non-destructive visual storytelling studio.

## Core Principles

- Visual controls for visual properties.
- Text controls for textual properties.
- Reusable components over one-off implementations.
- Editor UI should only contain controls that improve the active workflow.
- Metadata should not occupy permanent workspace.
- Architecture changes require explicit approval.
- Repository is the source of truth.

No live "Reader" surface exists anywhere in the codebase today (confirmed via repo-wide grep — only `FileReader` API hits and doc prose using "reader" to mean the end consumer of a downloaded export), confirming the request's own "eventually Reader" framing is correctly forward-looking, not a near-term target. The proposed strategy: build one new, additive, Studio-side adapter (`js/engineV2Adapter.js`, proposed not built) translating a resolved compiled theme into Engine V2's Scene Model shape — the missing reverse direction of `convergeSceneLayer`'s existing one-way translation — then migrate **consumer by consumer, never feature by feature** (the Decoration Shapes sprint's own "kept in lockstep by hand" duplication is cited as a live demonstration of what a feature-split produces at scale): Phase 0 builds and pixel-verifies the adapter against Museum Gallery with zero call sites wired to any real UI; Phase 1 migrates the lowest-stakes one-shot consumer (Card Designer's Frame swatch or `ThumbnailEngine.generate()`); Phase 2 moves the real export engine (`js/storyDestinations.js`, where page-furniture/Quote-composition parity actually has to close, not stay deferred); Phase 3 moves the live editor canvas only after full-fidelity parity is proven; Phase 4 retires Engine V1 only once every consumer has migrated and a full regression pass shows zero behaviour change; Phase 5 (Reader) is named as a future consumer of the same runtime, explicitly out of this document's own scope. Builder's Scene/Holder/Layer authoring model, the compiled `.vtheme` shape, `convergeSceneLayer`'s existing one-way translation, and every already-published Theme are explicitly named as staying completely frozen throughout — every proposed change is new Studio-side code (plus disclosed, scoped Engine V2 capability extensions for the ornament-preset/page-furniture gap, an explicit open question, not resolved by this document) reading from structures that already exist, never a change to how a Theme Author authors anything. Three real open risks are disclosed rather than glossed over: ornament-preset fidelity is sized as "the largest single fidelity gap" with no proposed solution yet; whether to extend `engineRuntime.js` with the missing drawing routines vs. keep a small permanent Engine-V1-only hybrid for page furniture is flagged as a real product decision needing explicit sign-off before Phase 2; and Engine V2's per-object drawing performance at editor-interaction frequency (every drag-move frame) was never measured and is flagged as needing a real performance pass before Phase 3, not assumed safe. No code was written — this is a strategy proposal only; per the standing "architecture changes require explicit approval" rule, even Phase 0's isolated, zero-call-site adapter would need its own sign-off before implementation begins.
